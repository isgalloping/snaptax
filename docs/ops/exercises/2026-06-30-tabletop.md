# Security Incident Tabletop Exercise — 2026-06-30

**Type:** Desktop simulation (no production changes)  
**Scenario:** Leaked `BLOB_READ_WRITE_TOKEN` in a public gist  
**Participants:** Engineering (IC) · Legal/Privacy (observer)  
**Duration:** 45 minutes

---

## Scenario narrative

A contractor accidentally commits a `.env.example` paste containing a valid **Production** `BLOB_READ_WRITE_TOKEN` to a public GitHub gist. A security researcher emails **snaptax.lightxforge@gmail.com** with subject **Security Report** 6 hours later.

**Assumed facts:**

- Token grants read/write to all receipt images in the production Blob store.
- Postgres credentials were **not** exposed.
- ~2,400 receipt images in Blob; ~800 active Ghost/Google users in the last 90 days.
- Researcher did not download bulk data; no evidence of exfiltration beyond proof-of-concept HEAD request.

## Decisions tested

| Question | Outcome |
|----------|---------|
| Rotate `GHOST_HMAC_SECRET` too? | **Yes** — precautionary; Ghost users must re-register |
| Notify all users or 90-day active only? | **90-day active** + anyone with receipts in exposed window |
| Rollback deploy? | **No** — app code unchanged; token rotation sufficient |
| GDPR supervisory authority? | Legal to assess Art. 33 — **likely yes** for EU users if read access confirmed |
